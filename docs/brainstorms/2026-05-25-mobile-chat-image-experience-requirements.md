# 手机端聊天出图体验：小酌陪你把回忆变成画面

## Summary

为 Drinking Time 新增移动端体验层：用户在手机上只和小酌聊天，聊着聊着小酌主动提议生成画面，图片在对话流里先以草图渐清晰的方式出现，用户通过左划（丢弃）、右划（收下）、点击局部编辑来反馈。收下的图自动绑定为剧本对应台词的故事版帧。手机端有两个页面：聊天页和剧本+故事版页（全能编辑）。

---

## Problem Frame

现有用户回忆和消化情绪的方式——发朋友圈、翻相册、跟朋友语音、写日记——都是单向的：朋友圈是表演给别人看，翻相册是自己闷着看，语音靠朋友有没有空，日记没有回应。没有一种方式是"随时在、会追问、还能把你模糊的画面变成看得见的图"的。

同时，Drinking Time 的 Creation Engine（桌面端）是为"导演/创作者"设计的专业工作台，对于"只想回忆不想操作"的普通人来说门槛太高——双面板布局、ShotTable、Script Viewer 这些概念对他们没有意义。这类用户需要的是：打开手机，跟一个懂他的人聊，画面自然就出来了。

目前整个产品没有移动端适配，没有手势交互，Creation Agent 也不会主动提议生成图片。

---

## Actors

- A1. **回忆者（用户）**：只想在手机上聊聊往事、看看画面的人。不关心镜头编号、分镜表、景别这些专业概念。操作方式是聊天、滑动、点按。
- A2. **小酌（Story Agent，扩展后）**：在手机端既采情绪样本，又能在时机合适时提议生成图片。是唯一的对话角色，不同界面提供不同能力。
- A3. **剧本+故事版页面**：手机端第二个页面，展示台词文字和每句下面对应的图，支持全能编辑。

---

## Key Flows

- F1. **聊天中自然出图**
  - **Trigger:** 用户和小酌聊到某段回忆，小酌判断情绪信号和场景描述足够具体。
  - **Actors:** A1, A2
  - **Steps:**
    1. 小酌在对话中主动说类似"我帮你画个画面？"
    2. 用户同意（发消息/点按钮）
    3. 对话区出现草图占位，画面从模糊逐渐变清晰
    4. 图片完全生成后，变为可滑动的卡片浮在对话上方
    5. 用户左划（丢弃）、右划（收下）、或点击进入局部编辑
  - **Outcome:** 用户在纯聊天场景中零操作门槛地获得一张情绪画面。
  - **Covered by:** R1, R2, R3, R4, R5

- F2. **左划丢弃 + 信号收集**
  - **Trigger:** 用户看到生成的图片，感觉不对。
  - **Actors:** A1, A2
  - **Steps:**
    1. 用户左划卡片
    2. 卡片飞出并消失
    3. 小酌追问"哪里不对？"或提供选项（氛围、人物、场景、颜色等）
    4. 用户回答后，小酌记录这个信号，优化下一次生成
  - **Outcome:** 系统收集了一条负反馈信号，下次生成更精准。
  - **Covered by:** R6, R7, R14

- F3. **局部编辑后右划收下**
  - **Trigger:** 用户觉得图片大体对，但某个局部不准确。
  - **Actors:** A1, A2
  - **Steps:**
    1. 用户点击/长按图片上某个区域
    2. AI 识别该区域的物体边界并高亮
    3. 用户描述要改成什么（打字或语音）
    4. 只重生该区域，其余保留，画面渐清晰
    5. 用户对修改满意后右划收下
  - **Outcome:** 用户在物体粒度上精准调整了画面，同时系统收集了编辑位置和意图作为信号。
  - **Covered by:** R8, R9, R10, R14

- F4. **右划收下 → 绑定到剧本**
  - **Trigger:** 用户右划一张满意的图片。
  - **Actors:** A1, A2, A3
  - **Steps:**
    1. 用户右划卡片
    2. 卡片飞入收藏动画
    3. 该图自动绑定到当前聊天上下文对应的剧本台词下面，成为该场景的故事版帧
    4. 用户切到剧本+故事版页面，可以看到图片已经出现在对应位置
  - **Outcome:** 用户的回忆从文字变成了有画面的故事版。
  - **Covered by:** R11, R12, R13

## Requirements

**手机端整体结构**
- R1. 手机端有且仅有两个页面：聊天页和剧本+故事版页，通过底部导航或类似方式切换。
- R2. 小酌是唯一的对话角色，贯穿手机端全流程。不引入新的 Agent 角色。
- R3. 手机端和桌面端共享同一套后端数据（卡片、剧本、图片、Shot 数据）。

**聊天页：对话与出图**
- R4. 小酌在聊天过程中根据情绪信号和场景描述的丰富度，主动提议"我帮你画个画面？"。用户同意后才生成。
- R5. 图片生成期间，先显示模糊/低分辨率的草图占位，逐步变清晰，让用户看着画面"长出来"。
- R6. 生成完成的图片以卡片形式浮在对话上方，支持左划（丢弃）和右划（收下）手势。
- R7. 左划丢弃后，小酌主动追问"哪里不对"来收集负反馈信号，用于优化后续生成。
- R8. 用户可以点击/长按图片上的某个区域，进入局部编辑模式。
- R9. 局部编辑时，AI 自动识别点击区域的物体边界并高亮显示，用户描述改动意图后只重生该区域。
- R10. 局部编辑完成后，用户可以继续编辑其他区域或右划收下。

**图片归属与故事版绑定**
- R11. 右划收下的图片自动绑定到当前聊天上下文对应的剧本台词，成为该场景的故事版帧（也是未来视频的首尾帧）。
- R12. 绑定关系由小酌从对话上下文推断，无需用户手动指定镜头编号。
- R13. 同一台词/场景下可以有多张图（版本迭代），最新收下的作为主图。

**信号收集**
- R14. 用户的每个动作——左划、右划、编辑位置、编辑描述——都作为信号记录，用于提升系统对该用户真实需求的理解精度。

**剧本+故事版页**
- R15. 展示剧本台词文字，每句台词下方显示对应的故事版图片。
- R16. 支持全能编辑：修改台词文字、调整场景顺序（拖拽）、点击图片进入局部编辑、删除场景。
- R17. 剧本+故事版页面的数据与聊天页实时同步。

## Acceptance Examples

- AE1. **Covers R4, R5.** Given 用户和小酌聊到"那天下午他走了之后我一个人坐在阳台上，太阳快下山了"，when 小酌判断场景+情绪足够具体并提议"我帮你画个画面？"且用户说好，then 对话区先出现一个模糊的画面（1-2秒内），然后逐步变清晰成为一张完整的图。

- AE2. **Covers R6, R7.** Given 一张图片已生成并以卡片形式浮在对话上方，when 用户左划该卡片，then 卡片飞出消失，小酌随后追问类似"是氛围不对、人不对、还是场景不对？"来收集反馈。

- AE3. **Covers R8, R9, R10.** Given 一张图已生成且画面里有一棵树，when 用户点击树的位置，then AI 识别出树的边界并高亮，用户输入"应该是一棵枯树"后该区域重新生成，其余画面保留不变，用户满意后右划收下。

- AE4. **Covers R11, R12, R13.** Given 用户正在聊关于"阳台上看夕阳"的回忆且右划收下了一张图，when 用户切到剧本+故事版页面，then 对应的台词下方已经出现这张图作为故事版帧。

- AE5. **Covers R15, R16.** Given 剧本+故事版页面已有三个场景各带一张图，when 用户拖拽第三个场景到第一个位置、修改第二个场景的台词、点击第一张图进入局部编辑，then 所有操作正常执行，顺序和内容更新实时反映。

## Success Criteria

- 一个"只想回忆不想操作"的用户，打开手机和小酌聊 10 分钟，能收下至少 1-2 张满意的情绪画面，全程不需要理解任何专业概念（镜头编号、景别、分镜表等）。
- 用户的每次左划、右划、局部编辑都被记录为信号，系统在同一次会话中的后续生成可见地更贴近用户的描述。
- 剧本+故事版页面让用户能像翻一本图文书一样浏览自己的回忆，并能自由调整。
- 手机端体验不干扰桌面端现有功能——两端共享数据但 UI 完全独立。

---

## Scope Boundaries

- 桌面端的改造或重新设计
- 视频生成或播放（图片是视频首尾帧的素材，但视频生成本身不在此范围）
- 社交分享功能（发朋友圈等）
- 多人协作
- 语音输入（可作为后续增强，v1 用文字输入）
- 图像版权、署名、商业使用合规审查
- Push 通知或后台提醒

---

## Key Decisions

- **小酌统一角色，不引入新 Agent。** 用户从头到尾只和一个角色聊天。手机端的小酌在现有情绪采样能力之上扩展出图能力，而不是拆成两个 Agent。理由：对于"只想回忆"的用户来说，面对两个角色会打破信任感和陪伴感。
- **图片在对话流中蹦出来，不跳转到另一个界面。** 聊天和出图在同一个空间发生。理由：跳转会打断回忆的情绪流；对话中自然出现才符合"聊着聊着画面就出来了"的体验预期。
- **左划/右划/编辑是信号收集手段，不仅是交互。** 每个用户动作都在帮系统理解这个人。理由：收集的信息越多，系统算出用户真正需求的概率越大。
- **先草图再精修的等待体验。** 不用加载动画或纯等待，而是让用户看着画面从模糊变清晰。理由：等待本身变成有仪式感的时刻，增强情感连接。
- **右划的图直接绑定为故事版帧。** 不需要用户手动归类。理由：对"不想操作"的用户来说，右划就是最终确认动作，系统应该自动完成后续归档。
- **手机端两个页面而非一个。** 聊天页 + 剧本故事版页。理由：聊天页负责回忆和生成，故事版页负责全局浏览和编辑。合在一起会让聊天页过于拥挤。

---

## Dependencies / Assumptions

- 复用现有的 fal.ai + Flux 图像生成和 SAM 2 分割能力，交互层面做移动端简化。
- 假设渐进式图片加载（草图→清晰）可以通过先生成低分辨率再 upscale，或 progressive JPEG/streaming 实现——具体方案留给 planning。
- 手机端小酌的"出图时机判断"需要扩展 Story Agent 的 system prompt，增加图像生成相关的工具调用能力——这是对现有 Story Agent 的能力扩展，不是新建 Agent。
- 假设手机端是移动端 Web（responsive），不是原生 App——留给 planning 确认。
- 桌面端现有的 Creation Agent + Creation Page 保持不变，手机端是独立的体验层。

## Outstanding Questions

### Deferred to Planning

- [Affects R1][Technical] 两个页面之间的导航形式（底部 tab bar、顶部切换、手势滑动切页等）。
- [Affects R4][Technical] 小酌判断"出图时机"的具体信号设计——需要什么程度的情绪+场景描述才触发提议。
- [Affects R5][Needs research] 草图→清晰的渐进加载具体实现方案（低分辨率先出 + upscale？progressive image？streaming？）。
- [Affects R6][Technical] 卡片滑动手势的技术方案（Framer Motion gesture？react-spring？原生 touch event？）。
- [Affects R11, R12][Technical] 小酌从对话上下文推断图片应该绑定到哪句台词的具体策略。
- [Affects R14][Technical] 信号数据（左划原因、编辑位置、编辑描述）的存储结构和如何反馈到生成模型。
- [Affects R3][Technical] 手机端和桌面端的数据同步机制——是否需要实时同步或按需加载。
- [Affects R15, R16][Technical] 剧本+故事版页面在手机端的布局设计（垂直滚动？卡片式？）。
