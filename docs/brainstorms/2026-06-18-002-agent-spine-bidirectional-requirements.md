# Agent 脊柱：贯穿创作全程的双向理解骨架

## Summary

把散落在各阶段、各管线里的"Agent 分析"重塑成一根贯穿故事创作全程的**脊柱**——它以"带出处的编辑账本"形式承载用户演进中的意图，随阶段改变 Agent 的职责，为每个决定附可回看的理由，并支持理解在骨架上**双向流动**；昂贵的出图/出视频作为脱钩的下游线，仅在用户显式确认时才渲染。

---

## Problem Frame

产品的领域逻辑本是一条直线：**聊天 → 图 + 提示词 → 视频**，中间由 Agent 读取上下文做分析，依据是聊天框内容与用户的手动修改。但代码没有顺着这条直线生长，导致"改一个东西无比艰难"，且改动速度在持续变慢——存在滑向"无法修改"的真实风险。

体检证据（当前状态，仓库相对路径）：

- `client/src/features/storyAgent/StoryAgentContext.tsx` 单文件 **2670 行 / 29 个 useState / 43 个 useCallback**，同时承担聊天、故事文档、出图、意图识别、存盘同步、选区编辑共六类职责。
- `client/src/features/storyAgent/views/` 下几乎每个面板都从同一个 `useStoryAgent` 取数据——5 个面板"共喝一缸水"，动一处则全体重渲染、互相牵连（用户称之为"串"）。
- 核心概念无归属：`prompt` 散落约 96 个文件、`shot` 约 82 个、`emotion` 约 62 个。
- `server/services/agentRuntime.ts` 仅 70 行、且四个 Agent 里只有 `creationAgent` 在用；`artAgent`/`scriptAgent`/`storyAgent.chat` 各调各的 LLM；`server/services/editContext.ts` 只在 `server/routers.ts` 一处被消费，未顺流程贯穿。
- 七个分析服务（`sceneAnalysis`/`semanticAnnotation`/`resonanceSignal`/`segmentation`/`renderGate`/`artDirection`/`composeScenePrompt`）各自独立触发，没有一个共享的"理解"对象把它们串起来。

**根因**：本该作为一等公民的"故事/创作的连续理解"在代码里不是一个独立的东西——它被困在前端 god-object、在七个分析服务里被各自重新推导、又被三条平行管线（桌面故事 / 手机 / 创作+分析）各养一份。每加一点 Agent 能力，都要手动在三条管线之间重新接线，成本即是用户感到的摩擦。god-object 不是病本身，而是"脊柱缺失"的症状——因为没有真正的脊柱，所有需要全局理解的东西只能往它身上堆。

## Actors

- A1. 用户/创作者：在不同阶段处于不同清晰度（从混沌到具体），保留最终的修改与剪辑权。
- A2. 阶段感知 Agent（同一循环、随阶段换职责）：在每个阶段判断"此刻最该被点破或被代劳的是什么"，给方案、附理由、吸收修改、带着往下走。
- A3. 渲染下游（出图线 / 出视频线）：脱钩的昂贵计算线，接收"过时"信号，仅在用户显式晋级时执行。

---

## Key Flows

- F1. 混沌起步
  - **触发：** 用户处于"不知道要做什么"的混沌状态。
  - **Actors：** A1, A2
  - **步骤：** 用户随意聊 → 意图识别替其定方向（"你其实想做的是 X"）→ 写入脊柱作为后续依据。
  - **Covered by：** R1, R4
- F2. 选区 + 面板 + 对话 修改（同一循环）
  - **触发：** 用户想改某处。
  - **Actors：** A1, A2
  - **步骤：** 选中一个目标（一个镜头 / 一句话 / 画面里一个元素）→ 当前所在面板提供"改的是哪类事"的上下文 → 用户在对话框口述怎么改 → Agent 执行并说明理由 → 修改作为带出处的事件写回脊柱。
  - **Covered by：** R5, R6, R7, R12
- F3. 向下投影 + 按钮晋级渲染（三线脱钩）
  - **触发：** 上游（聊天/剧本/图）内容更新。
  - **Actors：** A1, A3
  - **步骤：** 上游变化使下游被标记"已过时"，但不自动重算 → 用户决定是否花这笔渲染 → 按【重新渲染】出图；图更新后视频被标记"可能需替换"，用户确认"有必要换视频吗" → 按按钮才在独立的视频线渲染。
  - **Covered by：** R8, R9, R10
- F4. 向上反推（北极星，首版不建）
  - **触发：** 用户在下游做过手动修改（改视频提示词、抠图像像素）后，想回头改剧本。
  - **Actors：** A1, A2
  - **步骤：** Agent 回看脊柱账本里"用户在下游改过什么" → 反推剧本该如何调整以重新自洽 → 提建议给用户 → 用户点头后才谈下游重渲。
  - **Covered by：** R13；实现见 Scope Boundaries → Deferred for later。

## Requirements

**脊柱：连续理解 / 编辑账本**
- R1. 存在一份贯穿创作全程的共享理解（脊柱），承载：演进中的意图、当前阶段、每个决定的理由、用户的手动修改。
- R2. 脊柱以"带出处的编辑事件"为记录单位——每次编辑记录：发生在哪个阶段、改的是哪个目标、改成了什么、为什么。
- R3. 脊柱是唯一的跨阶段共享状态；各面板/各阶段只通过窄接口读写它，不再彼此直接共享一缸数据。

**阶段感知 Agent**
- R4. Agent 是阶段感知的：同一交互循环（读理解 → 判断此刻该点破/代劳什么 → 给方案+解释 → 吸收修改 → 带往下走），职责随阶段变化——混沌期定意图、剧本期指导表达、出图期翻译意图、分镜期预配镜头。
- R5. Agent 为每个决定附"为什么"，且该理由是挂在镜头/图等对象上、可持久化、可回看的一等数据，而非聊天里飘走的一句话。
- R6. Agent 会管住用户输入：能判断用户何时多说有用、何时已说够，并可提示"够了/不必再说"。

**修改交互：选区 + 面板 + 对话**
- R7. 修改通过"选区 + 当前面板 + 对话"完成：选中目标定位"改哪个"，当前面板提供"改的是哪类事"，对话提供"怎么改"。
- R12. 用户保留最终修改/剪辑权；其修改作为事件写回脊柱，成为下一步 Agent 的依据。

**渲染脱钩与晋级**
- R8. 三条线（聊天/理解、出图、出视频）刻意不自动级联：上游变化只将下游标记为"已过时"，不自动触发重算。
- R9. 昂贵渲染（出图、出视频）由用户显式触发——通过【重新渲染】/【替换视频】等按钮与确认完成。
- R10. 当图更新后，是否替换已生成的视频需用户确认；确认后才在独立的视频线渲染。

**面板架构**
- R11. 桌面端共 5 个面板，各对应一个创作方向、信息走脊柱而不互串；其中仅"故事聊天面板"贯穿始终。冗余的跨面板共享信息予以删除，使各面板接口单一。

**Canonical 故事一致性（Agent context 统一）**
- R14. **脊柱 = 唯一 canonical 故事 = Agent 的统一 context**。4 个创作面板（Story Cards / Script / 动态分镜 / 提示词表）是这唯一故事的**派生视图**，不是 4 份并行真相。Agent 永远读 canonical 故事，绝不读某个面板的残缺副本。（现状反例：cards/scripts 在内存、分镜/提示词表从服务端 body 再派生，4 份真相 2 个层级，会静默打架——这正是"Agent context 无法统一"的根因。）
- R15. **派生血缘 + 过时标记**：每个阶段产物记录"我派生自上游哪个版本"，上游变化时下游被**显式标记为"过时"**（而非静默不一致，也非自动重算）。"过时"是被看见、被记录的状态。
- R16. **同源但脱钩**：下游阶段（分镜、提示词、出图、出视频）不自动跟随上游重生，由用户按钮显式重生（与 R8/R9 渲染脱钩同一机制，扩展到全部创作阶段）。脱钩发生在"投影/重生"层，canonical 故事本身始终唯一、自洽。

**双向流动**
- R13. 理解可在骨架上双向流动：向下投影（如剧本→图）纳入首版地基；向上反推（下游手动修改→反推剧本）为北极星能力，首版不建但架构留位（见 Key Decisions D6）。

## Acceptance Examples

- AE1（Covers R8, R9）：用户在聊天里把主角从"老人"改成"少年"。出图线**不**自动重渲染；已有的图被标记"已过时"，直到用户按【重新渲染】才生成新图。
- AE2（Covers R10）：用户按【重新渲染】更新了某镜头的图。系统询问"有必要替换该镜头的视频吗"；用户不确认则视频保持不变，确认后才在视频线渲染。
- AE3（Covers R5, R12）：用户选中一个镜头、口述"再压抑一点"。Agent 调整并给出理由"降低了高光、收紧了构图，以呼应剧本里这一幕的失落感"；该修改与理由写回脊柱并可回看。
- AE4（Covers R6）：用户在意图已足够清晰后仍在补充细节。Agent 提示"信息已经足够开始出图了，可以先看效果再调"，而非continuing索取更多输入。

## Success Criteria

- 改"Agent 在某个阶段该做什么"只需改动**一个地方**（脊柱/对应阶段职责），不必在三条管线之间重新接线。
- 5 个面板各管一个方向、互不牵连；编辑一个面板不再引发其他面板的无谓重渲染（"不会串"）。
- 界面操作跟手、不卡——god-object 引发的"一动全抖"重渲染风暴消失（附带收益）。
- 脊柱作为"带出处的编辑账本"成立，使向下投影立即可用、向上反推日后可接入而无需重新架构。

---

## Scope Boundaries

**Deferred for later（暂缓，但架构留位）**
- 向上反推（像素/提示词的手动修改 → 反推剧本该怎么改）：北极星能力，依赖 AI 判断质量，需单独验证，首版不实现。
- 真正建出"视频线"的渲染实现：视频是一条一等的按需线，但首版只设计好脱钩边界与"过时/晋级"机制，不实现视频渲染。
- "面板即上下文"的完整实现（如在动态分镜面板里改=只调剪辑节奏）：首版不建，架构把"当前活动面板"作为一等上下文输入留好位。
- 三条平行管线（桌面/手机/创作+分析）的正式合并：是脊柱落地后的自然结果，不是第一刀。

**Outside this product's identity（与产品身份相悖）**
- 用按钮做"意图/理解"层面的编辑——意图编辑必须靠对话；按钮只用于触发昂贵渲染的显式晋级。
- 上游变化自动级联重渲染下游——刻意排除，以免每次聊天改动都偷偷烧钱重渲。
- 推倒重写整个项目。

## Key Decisions

- D1. **脊柱优先于拆 god-object**：先立脊柱（唯一共享理解），`StoryAgentContext` 的瘦身是结果而非目标——因为它不必再独自扛全局。
- D2. **脊柱 = 带出处的编辑账本**，而非一份扁平状态。出处（阶段/目标/改动/理由）是支撑双向流动与"可回看理由"的共同地基。
- D3. **理解层双向、连续、便宜；渲染层脱钩、按需、显式**。双向性住在理解脊柱里，不住在渲染线里。
- D4. **交互界线**：改意图/理解→对话、不加按钮；触发昂贵渲染（出图/出视频）→显式按钮+确认。两条原则不矛盾，互为边界。
- D5. **5 面板单一方向 + 唯一贯穿的聊天面板**；删除电脑端冗余的跨面板共享信息，使接口单一、不互串。
- D6. **第一刀（结构）**：把脊柱立成唯一共享状态 → 5 面板各收窄到一个方向、走窄接口 → 故事聊天成唯一贯穿入口 → 删冗余共享信息。**第一个行为证明**叠在结构之上：让"意图+理由"从剧本阶段无损贯穿到出图阶段。
- D7. **脊柱拥有 canonical 故事，服务端只是持久化（2026-06-18 修订，改写 D6 的所有权判断）**：4 面板从脊柱投影，服务端 body 不再是另一份竞争真相。这比"让 react-query 各管各的 body、脊柱只读"更进一步——后者治重渲染，治不了"4 份真相打架"。脊柱里从一开始就要带"阶段血缘 + 过时标记"字段（见 R14–R16），否则以后又得重构。**起因**：用户在界面上发现同一个故事 4 面板内容不统一，正确推出"Agent context 无法统一"。

## Dependencies / Assumptions

- 现有 `server/services/imageGen.ts` 是三条线唯一共享的底座，应作为脊柱落地时的复用点保留。
- 重构不改变 LLM 调用与图像生成本身的延迟——"出图/分析"本身不会因此变快。
- "消除重复 AI 调用"为潜在但**未验证**的提速来源：取决于当前是否存在跨阶段重复推导，需实测确认。

## Outstanding Questions

- 性能是否升格为显式目标？当前定位：主目标是"治改不动"，界面响应为附带收益，"出图耗时"为非目标，"消除重复 AI 调用"为待验证项。
- 5 个面板与"创作方向"的确切映射，以及哪些跨面板共享信息可安全删除，需对照现有 `client/src/features/storyAgent/views/` 逐一确认。
- 向上反推（R13）的验证方式：用什么样的小实验判断"从下游修改反推剧本"是否达到可用质量。
