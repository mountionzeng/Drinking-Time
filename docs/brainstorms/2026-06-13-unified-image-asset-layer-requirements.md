# 统一图片资产层：手机轻记录，电脑强生产

## Summary

建立手机端与电脑端共用的一套图片资产真相：手机端负责随时记录、轻量出图与选择，电脑端负责继承已选画面、管理镜头和版本、深度修改与生产。用户始终只和小酌交流，小酌在后台调度画面分析、美术判断、镜头与出图能力。

---

## Problem Frame

当前产品已经能从手机聊天、故事美术定调和 Creation Engine 多处生成图片，但这些图片使用不同的归属方式：有的绑定故事，有的绑定项目，有的使用数字镜号，有的使用 `SH01`，还有一类是美术方向候选。图片虽然进入了同一份存储记录，却没有形成同一套可跨页面理解的资产身份。

直接结果是用户明明已经生成、选择甚至修改过图片，进入 `/creation` 后却看不到。当前界面只展示被标为当前版本且镜号完全匹配的图片；旧版本、无镜号图片、数字镜号图片和美术候选都可能变成“存在于数据里，但不在任何界面里”的孤立资产。部分更早的记录还只剩元数据，图片文件本身已经失效。

与此同时，手机和电脑承担的任务不同。手机的价值是随时打开、记录一刻、上传照片、聊两句、快速判断喜欢或不喜欢；电脑的价值是把已经积累的故事和画面变成可控制的生产工程。若两端复制同一套复杂界面，手机会过重；若两端继续各存各的，电脑就无法继承用户在手机上已经完成的判断。

---

## Actors

- A1. 用户：在手机上随时记录，在电脑上完成深度视觉生产。
- A2. 小酌：唯一对用户可见的对话角色，理解用户意图并协调内部专业能力。
- A3. 手机记录端：承载聊天、照片输入、轻量候选、喜欢/淘汰和简单修改。
- A4. 电脑 Creation Engine：承载镜头级主图、版本、比较、重绑、局部修改和专业生产。
- A5. 内部专业 Agent：包括画面分析、美术判断、镜头理解、出图和局部修改能力，不作为额外人格要求用户切换。
- A6. 统一图片资产层：记录每张图片属于哪个用户、故事时刻、故事、项目、镜头和版本，并保存用户选择信号。

---

## Key Flows

- F1. 手机随时记录并收下一张图
  - **Trigger:** 用户在手机端记录一段经历、上传照片或要求把这一刻画出来。
  - **Actors:** A1, A2, A3, A5, A6
  - **Steps:** 小酌理解当前故事时刻；后台能力结合照片、对话和美术方向生成候选；用户喜欢、淘汰或简单修改；被喜欢或确认的图片成为已选资产，并与当前故事时刻关联。
  - **Outcome:** 用户不需要理解镜头编号，也能留下可供后续生产继承的画面判断。
  - **Covered by:** R1, R4, R5, R6, R9, R13

- F2. 电脑继承已选画面进入镜头生产
  - **Trigger:** 用户打开同一项目的 `/creation`。
  - **Actors:** A1, A2, A4, A6
  - **Steps:** 系统加载项目的镜头和统一图片资产；把已明确绑定镜头的主图直接放入对应镜头；把只绑定故事时刻的已选图按来源卡片或场景关系映射到镜头；无法确定的图片进入待归属区，不静默丢失；用户可以确认或重绑。
  - **Outcome:** 手机上已经选好的图自然成为电脑生产的起点，而不是重新从空白开始。
  - **Covered by:** R2, R7, R8, R10, R11, R12

- F3. 通过小酌修改当前镜头
  - **Trigger:** 用户在 `/creation` 选中镜头或图片后，在小酌聊天框描述修改。
  - **Actors:** A1, A2, A4, A5, A6
  - **Steps:** 小酌读取当前镜头、主图、历史版本、故事材料和美术方向；判断用户是在分析、改提示词、生成变体、局部修改还是更换镜头归属；调用对应专业能力；生成的新版本进入同一资产版本链，默认成为待确认版本；用户确认后更新主图。
  - **Outcome:** 用户保持一条连续对话，不需要知道后台由哪个 Agent 完成任务。
  - **Covered by:** R3, R14, R15, R16, R17

- F4. 查看历史与处理失效资产
  - **Trigger:** 用户在电脑端查看某镜头的图片历史或系统遇到旧数据。
  - **Actors:** A1, A4, A6
  - **Steps:** 系统展示当前主图、可用历史版本、淘汰版本和缺失文件记录；可用旧版本可以恢复为主图；文件已丢失的记录显示明确状态和原有上下文，但不伪造图片或自动重绘。
  - **Outcome:** 图片不会因为被替代或格式不兼容而消失，真实丢失也不会被掩盖。
  - **Covered by:** R10, R11, R12, R18

## Requirements

**统一资产身份与归属**

- R1. 手机端和电脑端必须读写同一套图片资产记录，不再维护彼此独立的图片真相。
- R2. 每张图片可以关联用户、故事、项目、故事时刻或卡片、镜头及父版本；不是所有关联都必须在生成当下齐全。
- R3. 图片必须保留版本链、生成来源、当前状态和用户反馈，使生成、精修、局部修改和恢复旧版可追溯。
- R4. 手机端不要求用户理解或选择镜头编号；图片可以先绑定故事时刻，待剧本和镜头形成后再映射。
- R5. 用户右划、明确喜欢或确认正式版时，图片成为“已选资产”；仅生成成功不等于用户已经选中。
- R6. 淘汰、未选和比较候选保留为审美反馈与历史资产，但不能自动成为镜头主图。

**手机端：轻记录**

- R7. 手机端聚焦聊天、照片输入、快速出图、喜欢/淘汰、确认正式版和简单修改，不展示专业镜头表、完整提示词或复杂版本树。
- R8. 手机端默认只展示当前候选、正在处理的图片和已收下结果；历史细节留给电脑端。
- R9. 手机上的所有选择和修改必须及时写入统一资产层，使电脑端无需依赖同一浏览器的本地状态。

**电脑端：强生产**

- R10. `/creation` 必须显示当前项目下所有与生产有关的图片，包括镜头主图、历史版本、待确认版本和无法自动归属的图片。
- R11. `/creation` 必须兼容既有数字镜号、标准镜号和无镜号资产，并将可确定的旧资产迁移或映射到统一镜头身份。
- R12. 已选图片进入 `/creation` 时应优先成为对应镜头的主图；映射有歧义时进入待归属区，由用户确认，不得静默隐藏。
- R13. 电脑端支持查看版本历史、比较候选、恢复旧版本、重新绑定镜头、生成变体和进入局部修改。

**小酌与后台 Agent 协作**

- R14. 小酌是手机和电脑唯一可见的对话角色，用户不需要切换或点名其他 Agent。
- R15. `/creation` 的小酌必须读取当前焦点镜头、主图和版本历史，以及相应故事材料、美术方向和用户反馈。
- R16. 小酌必须能识别并执行画面分析、提示词调整、整图变体、局部修改、镜头重绑和版本恢复等意图。
- R17. 专业 Agent 的分析和执行结果由小酌用统一口吻解释；后台分工不能造成多套互相冲突的对话状态。

**兼容与真实性**

- R18. 对元数据存在但图片文件已失效的历史资产，界面必须显示“文件缺失”及可用上下文；本轮不自动重绘，也不假装原图仍可恢复。

## Acceptance Examples

- AE1. **Covers R1, R4, R5, R9, R12.** Given 用户在手机上为一段故事右划收下一张图片，且当时只有数字场景编号，when 用户之后在电脑打开同一项目的 `/creation`，then 该图片出现在可对应的镜头中并成为优先主图，用户无需再次上传或选择。
- AE2. **Covers R6, R10, R13.** Given 同一镜头先后生成三张图，其中一张被收下、一张被淘汰、一张尚未确认，when 用户在电脑查看该镜头，then 主图、历史和待确认状态清楚区分，三张记录都可追溯。
- AE3. **Covers R11, R12.** Given 旧图片使用 `"2"` 作为镜号，而项目镜头使用 `SH02`，when `/creation` 加载，then 系统识别它们为同一镜头；若项目不存在第二镜，则图片进入待归属区而不是消失。
- AE4. **Covers R14, R15, R16, R17.** Given 用户在 `/creation` 选中 SH02 的主图并对小酌说“人物不要看镜头，窗外再亮一点”，when 小酌处理该请求，then 后台完成所需画面分析和修改，产生 SH02 的新版本，并由小酌说明修改结果。
- AE5. **Covers R10, R13.** Given 用户曾为 SH01 生成多个版本且当前主图不是最早版本，when 用户打开历史并选择恢复最早版本，then 该版本成为新的主图，原主图仍保留在版本历史中。
- AE6. **Covers R18.** Given 一条旧图片记录指向已经不存在的本地文件，when 用户在电脑查看相关资产，then 界面显示缺失状态、原提示和归属信息，不展示破图，也不声称已恢复。
- AE7. **Covers R7, R8.** Given 同一故事积累了十余张候选和多个版本，when 用户只在手机端继续记录，then 界面仍以聊天和当前候选为主，不要求用户管理完整版本树。

## Success Criteria

- 用户在手机上收下的图片，进入电脑端后可以直接继续生产，不需要重新寻找、上传或解释。
- `/creation` 不再出现“数据库有图片、界面完全看不到”的情况；可归属、待归属、历史和缺失状态都有明确去处。
- 手机端保持轻量，用户可以在短时间内完成记录、出图和选择；电脑端具备镜头级生产与版本管理能力。
- 用户始终感受到自己在和小酌协作，后台 Agent 分工提升能力而不增加认知负担。
- 实施规划不需要再发明双端职责、图片选择语义、历史图处理或 Agent 可见性规则。

---

## Scope Boundaries

- 本轮不包含视频生成、剪辑、运镜、配音或成片导出。
- 不把手机端扩展成专业镜头表、复杂提示词编辑器或完整版本管理器。
- 不做 Photoshop 级图层、画笔和手工蒙版系统。
- 不自动重建已经物理丢失的历史图片。
- 不建立跨用户共享或跨项目长期训练的全局审美模型。
- 不新增需要用户认识和切换的美术 Agent、镜头 Agent 或出图 Agent 人格。

---

## Key Decisions

- **一套资产真相，两种工作方式。** 手机和电脑共享数据与选择结果，但各自优化不同任务；共享不等于复制界面。
- **先绑定故事时刻，再补齐镜头归属。** 记录发生在生产结构之前，不能为了数据整齐而要求手机用户先理解镜头。
- **用户选择决定主图，不以生成成功代替审美确认。** 喜欢、确认和恢复才改变主图；候选和淘汰图保留但不越权。
- **小酌是统一入口，其他 Agent 是内部专业工具。** 连续关系比暴露技术组织结构更重要。
- **历史资产必须有去处。** 能映射的进入镜头，不能确定的进入待归属区，文件缺失的明确标记；不再静默过滤。

---

## Dependencies / Assumptions

- 当前故事、项目、镜头和图片记录可以作为统一资产层的迁移基础。
- 手机端和电脑端使用同一登录用户与服务端数据源。
- 现有画面分析、美术方向、出图、局部修改和镜头提示词能力可以继续复用，由小酌负责协调。
- 本地开发中的旧图片文件并非全部存在，因此兼容工作必须区分“记录可迁移”和“字节可恢复”。

## Outstanding Questions

### Deferred to Planning

- [Affects R2, R4, R11][Technical] 故事时刻、卡片、场景和镜头之间采用哪一种稳定映射键，才能兼容现有数据并避免后续再次依赖展示编号。
- [Affects R3, R5, R6][Technical] 主图、待确认、淘汰和缺失状态如何落在现有版本与反馈记录上，并保持迁移兼容。
- [Affects R15, R16][Technical] 小酌调度专业 Agent 时采用怎样的动作协议，既能覆盖分析和执行，又不复制多个对话状态。
- [Affects R10, R13][Technical] 电脑端图片资产区与现有 Shot Production Table 如何组合，才能提供强生产能力而不让表格进一步横向拥挤。
