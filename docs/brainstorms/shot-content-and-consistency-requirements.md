# 出图内容正确性 + 跨镜头一致性 — 需求文档

- **日期**: 2026-06-17
- **范围**: Standard（feature，两个有序阶段 / 共享分析内核）
- **状态**: 方向已对齐，待 ce-plan
- **合并自**: 原 `creation-editor-cross-shot-consistency-requirements.md` 与 `draw-this-moment-scene-analysis-requirements.md`（已合并，两文档作废）

## 问题

出图有两层都不对：

1. **内容不对**：按「把这一刻画出来」时，图只追求好看，没结合聊天内容分析"这一刻到底该呈现什么"——出的图不是用户想要的。
2. **不一致**：在 Creation 剪辑器里逐镜头出图，人物长相和美术风格跨镜头对不上，每次像重新生成。

**核心洞察（决定了做法和顺序）**：内容正确性比一致性更底层——把内容不对的图跨镜头锁得再一致也没价值（"一致的错图"=零价值）。而且两件事**共享同一个分析内核**：剪辑器"默认不画人、需要固定人物才引导上传照片"里的**"需不需要固定人物"检测，本身就是画面分析的一个输出字段**。所以合并、且**先内容、后一致**。

## 现状（已读代码确认）

**内容侧（storyAgent `DrawThisMomentPanel.tsx`）**：现有"分层架构"（`:99-169`）画面主体取自选中卡片内容（`cardHint`），**故意"不让 LLM 从对话瞎猜"**（`:100`）；最近 16 条 `history` 已传服务端但仅作补充，不决定画什么主体。→ 内容不由对话语境驱动。

**一致性侧**：服务端机制本身能用——`generateForMobile`（`server/routers.ts:1508-1661`）按 storyId 拉 `artDirection`，经 `characterReferenceOf`（`shared/artDirection.ts:262`）取人物参照、`toPublicImageUrl` 上传公网后注入 `--cref/--cw`；`imageGen.ts:278-287` 支持 `--cref`（锁人物）与 `--sref`（锁画风）。但断在两处：

1. 新剪辑器没有设参照入口（"设为主角"只在旧 `StoryCardsBoard`），`artDirection.references` 恒为 `[]`（实测 `.webdev/local-persist.json`：169 个 references 全空、0 个 `role:"character"`）→ 不加 `--cref` → 人物每镜头重生成。
2. `--sref` 接口没接上：`imageGen.ts:285` 支持 `styleRef→--sref`，但 `generateForMobile`（`:1620-1632`）从不传 `styleRef`，画风只靠提示词文本漂移。

## 总体原则

- **不新增 UI 按钮。** 画面分析确认、设锚点、引导上传照片等步骤一律走**对话形式**（Agent 在聊天里问/收/确认）。已有旧按钮（如 StoryCardsBoard「设为主角」）保留，新能力不走按钮。见记忆 prefers-conversation-over-buttons。
- **共享分析内核。** 一个服务端结构化画面分析层，同时喂"把这一刻画出来"和剪辑器两条线，两者都走 `generateForMobile`。
- **复用已验证的一致性服务端逻辑**（`characterReferenceOf`/`toPublicImageUrl`/`--cref` 注入），不重造。

## 阶段一：结构化画面分析内核（内容正确性 + 角色检测）

**目标**：让出图内容由对话语境驱动且可确认，并产出后续阶段需要的"是否需要固定人物"信号。

- 出图前，读最近对话 + 选中卡片，做**结构化画面分析**，得出"这一刻该呈现什么"：主体、是否为人物、是否反复出镜的角色、动作、情绪、关键视觉元素。
- **先在对话里向用户确认**（"我理解这一刻要呈现 X，对吗？"），用户可在对话里纠偏，确认后再出图。
- **默认克制**：分析判定不需要固定人物时，镜头默认画无人/空镜，不让 MJ 凭空造脸（当前 `creationAgent.ts:199` 自动铺镜头表无此克制，需补）。
- 避免重蹈"瞎猜"：分析是结构化的（明确字段）+ 有用户确认闸，不是拿模糊猜测直接出图。

**阶段一成功标准**
- 出的图内容明显贴合最近对话语境，而非泛泛好看图。
- 分析结果出图前以对话形式确认，用户可纠偏。
- 不需要固定人物的镜头默认空镜，不出现凭空人脸。
- 不新增按钮。

## 阶段二：在正确内容上叠跨镜头一致性

**前置**：阶段一交付后才做——此时锁的是用户已认可的画面，一致性才有意义，真出图验证也才不白烧额度。

- **人物锚点（对话设定）**：在对话里把某张满意图，或用户照片重绘图，设为人物锚点，写入 `artDirection.references` 的 `role:'character'`，复用既有 `--cref` 注入路径。
- **照片 → 重绘 → 锚点**：复用 `originalImageUrl`→`editMobileImage` 引擎（`routers.ts:1515,1578`），把用户照片结合当前画风重绘成**风格化人物图**，Agent 对话确认后设为锚点。**设为锚点的是风格化图，不是原始真人照**（原照只作重绘输入）。现状缺口：用户照片目前只当单镜头垫图，不持久化成锚点。
- **画风锁**：`generateForMobile` 接上 `styleRef → --sref`；锚点图同时充当 `--cref`（人物）和 `--sref`（画风）参照。

**阶段二成功标准**
- 设一张锚点后，后续镜头人物长相明显跨镜头锁定（不再每次重生成）。
- 画风跨镜头明显收敛（`--sref` 生效）。
- 剪辑器渲染仍走 `generateForMobile`（`CreationEditorContext.tsx:219,320`）。
- 用户被对话引导上传人物照后，照片经画风重绘并设为持久锚点，后续镜头自动锁该人物。

## 阶段间接口契约（语义层，ce-plan 须硬化成精确签名）

阶段一产出一个**结构化画面分析结果**，阶段二消费其中字段——这是两阶段的衔接点，必须钉死避免"接口对不上"：

- 分析结果须至少暴露：`主体描述`、`是否人物`、`是否反复出镜角色（角色标识/名 | 无）`、`动作`、`情绪`、`关键元素`、`是否需要人物锚点`。
- 阶段二只读 `是否需要人物锚点` / `反复出镜角色标识` 来驱动锚点逻辑；不反向依赖阶段二的字段。
- 该结果同时驱动出图 prompt 组装（喂 `generateForMobile`）。

> 具体的 TypeScript 类型/字段名/端点形状由 ce-plan 写死（见下方 Codex 约束），本文档只定语义契约。
