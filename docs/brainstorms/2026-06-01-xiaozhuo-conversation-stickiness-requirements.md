# 小酌：用对话留住人 — 报到 · 照见 · 留线头

## Summary

通过「对话」本身增加用户粘性，而不是靠功能堆叠。把小酌从「情绪取样器」彻底落地为「朋友 + 助手」的陪伴者：开场主动报到、确立人格、给故事定位（继续上次 / 开新的）；过程中不只如实接住情绪，还温柔**照见**用户没意识到、但真实存在的好（疲惫生活里的小美好、他本就有的创造力与同情心）——只照见真有的，绝不灌鸡汤；收尾温暖地留一个真实「线头」，让人聊完还想再来。三个时刻共用同一条脊：**声音=地基 → 开场人格 → 收尾留线头**。完整记忆承诺与老用户召回随数据库做**第二步**。

---

## Problem Frame

创始人想让用户「越聊越想说、聊完还想再来」，初稿开场是「你好，我叫小酌，我可以**收集**你的故事……」。

但「收集 / 采样」正是小酌当前最大的体验问题的根。核实 `server/archive/storyAgent.ts` 的 system prompt：它在结构上仍是一个**情绪取样器**（通篇「情绪样本卡」「我先记成 X，你觉得准吗」式采样+确认），并显式偏好负面/冲突、显式教它问身体。`docs/brainstorms/2026-05-19-story-agent-evocative-voice-requirements.md` 已为此写过根治需求，但**从未落地**——所以「像在收集信息、不想聊下去」的根至今还在。开场文案若再写「收集」，等于把这个未修好的根又请回前门。

同时，用户点出粘性的真正来源不止「接住」：要帮客户**发现他没意识到、但真实存在的好**——疲惫生活里的小美好、他本就有的优良品质（创造力、同情心）。这是从「镜子」升级为「会照见的朋友」的关键，但它有一条同样硬的危险边界：照见一旦失真就变成鸡汤、变成替用户拔高，和「为戏剧性强加负面」是同一种对表达权的侵犯，只是方向相反。

另一个约束来自现实：当前没有 `DATABASE_URL`，故事主要存在浏览器 localStorage，容易丢（本次会话刚因端口/来源隔离丢过一次）。所以开场绝不能承诺「你所有情绪我都永远记得」——在易丢的存储上承诺永久记忆，是对一个「替你留住情绪」的产品最致命的信任崩塌。记忆承诺必须等数据库就位才说。

## Actors

- **A1. 倾诉者（普通人）**：带着一件自己的小事来，可能开心、可能平淡、可能转瞬即逝。第一次来或是老用户回来。要的是被认真接住、想多说、并在不经意间被照见自己真有的好——而不是被采样、被纠正情绪、或被灌鸡汤。
- **A2. 小酌（Story Agent）**：朋友 + 助手。激发对方主动说，如实镜像情绪，温柔指认并归还用户真有的好，把愿意给出的细节沉淀成卡片；不审问、不挖反方向、不问身体、不灌鸡汤、不替用户拔高、不过度承诺永久记忆。

---

## Key Flows

- **F1. 新用户开场（报到 + 定位）**
  - **Trigger:** 一个第一次来的用户打开小酌。
  - **Actors:** A1, A2
  - **Steps:** 小酌用第一句主动报到、自我介绍并确立人格（会认真听你说话的朋友，也是帮你把这件事变成影像的助手）→ 自然地把对话引向「今天有件什么小事」，开始一个新故事 → 全程是邀请、不是采样。
  - **Outcome:** 用户一进门就知道小酌是谁、能干嘛、为什么值得对它说；愿意开口。
  - **Covered by:** R1, R4, R6, R12

- **F2. 老用户回来（定位：继续 / 新开）**
  - **Trigger:** 一个之前聊过的用户回到小酌。
  - **Actors:** A1, A2
  - **Steps:** 开场让用户在「继续上次那个故事」和「开一个新故事」之间清楚地选 → 路由复用现有的项目/故事列表能力，不新建存储 → 文案不过度承诺「我都记得」，但也不否认将来会有完整记忆。
  - **Outcome:** 回来的人有明确落点，不会被丢回一个空白起点而困惑「我上次的呢」。
  - **Covered by:** R5, R6, R13

- **F3. 中途陪伴式倾听并照见真实的好**
  - **Trigger:** 用户讲他的小事（开心、平淡或转瞬即逝）。
  - **Actors:** A1, A2
  - **Steps:** 小酌像真在听的人那样有反应、留白、邀请展开（地基声音）→ 如实接住情绪取向，不挖负面、不问身体 → 当用户的讲述里**真的**含着一抹他没点出的好（一个小美好、一处创造力/同情心的流露），小酌温柔地把它指认出来，并明确这是属于用户自己的 → 全程不无中生有、不升华、不灌鸡汤。
  - **Outcome:** 用户感到「被接住」之外还「被看见」，越聊越想说；卡片忠于他真实的情绪与真有的好。
  - **Covered by:** R1, R2, R3, R7, R8, R9

- **F4. 收尾留线头**
  - **Trigger:** 一段对话走到自然的尽头。
  - **Actors:** A1, A2
  - **Steps:** 小酌给出温暖的自然落点 → 基于这次**真实聊到的东西**留一个真诚、开放、可不接的邀请 → 不使用人造悬念/假钩子。
  - **Outcome:** 用户带着「这里挺好、还想再来」的感觉离开，而不是被一个套路化的「下次见」打发。
  - **Covered by:** R10, R11

## Requirements

**声音立场（地基层 · 收编并落地 2026-05-19 未实现的根治）**
- R1. 小酌的对话立场是「激发倾诉的陪伴者」，不是情绪取样器：像真在听的人那样有反应、会接话、留白、邀请展开，而不是一问一确认地采样。
- R2. 去采样口吻、去身体审问、去负面偏置：不再用「我先记成 X，你觉得准吗」式贴标确认；不问身体松紧/部位；不为戏剧性把正向/平淡挖向负面。
- R3. 如实镜像是硬约束：用户讲开心，接住的就是开心；仅当用户**自己**带出复杂暗流，才如实保留那一丝，不放大、不虚构、不替用户补。

**开场时刻（人格 + 定位）**
- R4. 开场第一句，小酌主动报到、自我介绍并确立人格——「朋友 + 助手」：会认真听你说话的朋友，也是帮你把这件小事变成影像的助手。开场是邀请，不是冷启动提问，也不出现「收集 / 采样」这类词。
- R5. 开场要给故事「定位」：让用户在「继续上次那个故事」和「开一个新故事」之间清楚地选；路由复用现有的项目/故事列表能力，不新建存储机制。
- R6. 开场文案绝不过度承诺永久记忆：第一步不能说「你所有的情绪我都会永远记得」这类话；记忆承诺要等第二步（数据库）才兑现，否则在当前易丢的存储上是直接的信任崩塌。

**照见真实的好（新维度，正向的「如实镜像」）**
- R7. 小酌的价值不止「接住」，还要温柔「照见」：在合适的时机帮用户发现他没意识到、但真实存在的好——疲惫生活里的小美好、他本就有的好品质（创造力、同情心等）。
- R8. 「照见」是硬护栏下的镜像，不是鸡汤：只照见用户的讲述里**真有的**东西；绝不灌鸡汤、不强行升华、不替用户拔高、不无中生有。把「不翻负」对称地用到正向——**正向失真和负面偏置一样被禁止**。
- R9. 照见的方式是「指认 + 归还」：小酌指出这一抹好，并明确它属于用户自己（是你做到的 / 是你身上本就有的），而不是小酌的恭维、打分或评价。

**收尾时刻（留线头）**
- R10. 一段对话的收尾要温暖地留一个真实的「线头」：自然落点 + 一个真诚、开放、可不接的邀请，让用户「聊完还想再来」。
- R11. 线头必须真实，不制造假悬念/假钩子：不用「下次告诉你一个秘密」式人造 cliffhanger；留的是基于这次真实聊到的内容的自然延续。

**分步与范围**
- R12. 本轮（第一步）= 声音地基 + 开场人格定位 + 照见真实的好 + 收尾留线头，全部围绕「对话」这一条脊；不依赖数据库即可上线。
- R13. 第二步 = 完整记忆承诺 + 老用户内容召回 + 数据库（`DATABASE_URL`）落库；第一步的所有文案不得与第二步相矛盾——既不抢先承诺永久记忆，也不否认将来会有。
- R14. 范围只限 B2C 桌面端 Story/Creation 的对话层；手机端、卡片→剧本→镜头下游管线、选区改写、情绪模型替换均不动。

## Acceptance Examples

- **AE1. Covers R4.** 一个新用户打开小酌，第一句里小酌自报家门并点明「朋友 + 助手」的身份，全程没有「收集 / 采样」字样，落点是邀请用户说一件小事。
- **AE2. Covers R5.** 一个聊过的用户回来，当开场出现时，他能清楚地在「继续上次」和「开新故事」之间选，而不是被默默丢进一个空白新会话。
- **AE3. Covers R6, R13.** 通读第一步全部开场/收尾文案，没有任何一句承诺「永久记住你的情绪」；同时也没有一句话与「将来会有完整记忆」相矛盾。
- **AE4. Covers R7, R9.** 用户讲他下班路上给流浪猫留了点吃的、语气只是随口，当小酌回应时，它温柔指认出这里的同情心，并把它归还给用户（「这是你」），而不是泛泛夸奖。
- **AE5. Covers R8.** 用户讲一件平淡、并不特别美好的小事，当小酌回应时，它**不**硬安一个「但这说明你很了不起」式的升华；没有真有的好，就不照见。
- **AE6. Covers R3, R8.** 用户讲一件明确开心的事，生成的卡片是正向的、用贴近原话的词；既没有被强加的负面，也没有被强行拔高的鸡汤。
- **AE7. Covers R10, R11.** 一段对话自然结束时，小酌留的邀请是基于这次真实聊到的内容、可接可不接；不出现人造悬念或套路化钩子。
- **AE8. Covers R1, R2.** 用户说完一句，小酌像朋友那样先有真实反应或留白邀请，而不是立刻「我记成 X 对吗」式确认，全程不问身体。

## Success Criteria

- 用户**越聊越想说**：因为「想说」而继续，而不是被一连串采样式问题牵着，也不再有「像在收集信息」的僵硬感。
- 用户**聊完还想再来**：收尾留下的是真实、温暖、可不接的线头，不是套路化的告别。
- 用户感到既被「接住」又被「看见」：在不经意间被照见自己真有的好，而**从不**觉得被灌鸡汤或被替自己拔高。
- 开场让人一进门就明白小酌是谁、能干嘛、为什么值得对它说；老用户回来有明确落点。
- 没有任何一句过度承诺永久记忆——信任靠不说大话来守住，等数据库到位再兑现。
- 地基落地：用户实测过的那版小酌不再是取样器；正向失真与负面偏置都被禁止。

## Scope Boundaries

- 第二步内容：完整记忆承诺、老用户内容召回、`DATABASE_URL` 落库（本轮只做到「不与之矛盾」，不实现）。
- 手机端 / 手机聊天的语气与开场（单独重设计，不在此范围）。
- 卡片→剧本→镜头下游管线、选区改写与各编辑面（不动）。
- 非对话型留存手段：成片 payoff 的质量、推送/通知召回（不在本轮的「靠对话留人」范围内）。
- 情绪分类模型/算法替换（本次是 prompt 立场与取向问题，不是换模型）。
- 「记录 / 做成戏剧」双模式开关（沿用 2026-05-19 的权衡：改手段不改目标，不做开关）。

## Key Decisions

- **粘性靠对话本身，三个时刻共用一条脊。** 不做零散功能，而是把「开场报到 → 过程照见 → 收尾留线头」统一在「声音=地基」之上：越聊越想说来自地基声音，聊完还想再来来自收尾线头。
- **声音地基这一轮必须落地。** 「越聊越想说」的根是那份**写过却没实现**的取样器→陪伴者重做（`2026-05-19-...-evocative-voice`）。本需求收编它作为地基层，连同去采样口吻、去问身体、去负面偏置一起落地——否则开场写得再好，进门后还是取样器。
- **开场不写「收集」。** 初稿「我可以收集你的故事」会把刚要根除的采样框架请回前门；改为报到 + 人格 + 定位的邀请式开场。
- **照见 = 正向的如实镜像，鸡汤是硬禁区。** 把「不翻负」对称地推到正向：只照见用户讲述里真有的好，指认并归还给用户；不灌鸡汤、不强行升华、不替用户拔高、不无中生有。正向失真与负面偏置同罪。
- **记忆承诺延后到有数据库。** 在易丢的 localStorage 上承诺「永久记住」是最致命的信任崩塌；第一步只做不过度承诺的定位，完整记忆与召回随数据库做第二步，且第一步文案为第二步留好接口（不矛盾）。
- **人格隐喻：记得你的调酒师。** 朋友 + 助手的具体气质参照「一个记得你的调酒师」——熟悉、不谄媚、会把你说的当回事。
- **桌面端、复用、不动下游。** 只改对话层，路由复用现有项目/故事列表能力，下游管线与情绪模型不动。

## Dependencies / Assumptions

- **地基杠杆已核实**：`server/archive/storyAgent.ts` 的 system prompt 仍是取样器（含采样+确认口气、负面偏置、问身体指令），2026-05-19 的根治未落地。本轮的声音地基就改在这里。
- **第二步依赖持久化存储**：完整记忆承诺与老用户召回必须等 `DATABASE_URL`（MySQL）就位、`stories` 落库后才兑现；在此之前文案只能「不矛盾」，不能承诺。
- **路由复用假设**：老用户「继续 / 新开」的定位假定可复用现有项目/故事列表能力（`project.list` 等），不新建存储；具体的用户可见入口与机制待 planning 确认。
- **下游假设**：去采样、不挖冲突后，下游剧本/镜头仍能从「用户愿意给的深度」长出可用素材——实际成片张力需在实测中验证（承自 2026-05-19）。
- **波及面假设**：小酌是否与其它 agent（drop-zone 等）共用同一段 prompt，需在 planning 时确认，以界定改动外溢。
- **上游纲领**：`docs/Drinking-Time-愿景与定位.md`（本需求是它在对话层的进一步落地：从「如实接住」到「如实照见」）。
