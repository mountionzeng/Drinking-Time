# Drinking Time 欢迎页纳音与老黄历改造需求

## Summary

本需求定义一个“两层结构”的 Drinking Time 欢迎页：首屏保留两个核心开始入口，同时让页面视觉随当天纳音五行变化；展开层展示农历、纳音和真实老黄历 API 信息。第一版应保留用户提供的 standalone HTML 参考稿中的插画与动画方向，只做产品适配，不做大幅美术重绘。

---

## Problem Frame

当前欢迎页的两个入口足够直接，但仪式感和“今天打开 Drinking Time 的独特气息”还不够强。用户希望欢迎页每天有变化，变化依据不是随机装饰，而是带有东方日历感的纳音五行、农历和老黄历信息。

同时，这个页面仍然承担明确的产品入口职责。用户打开页面时，应该先知道自己可以从“上传素材”或“聊故事”开始，而不是先被大量黄历信息淹没。参考稿里的插画和动画已经建立了合适的五行饮品氛围，因此本次改造应继承它，而不是重新发明一套视觉语言。

---

## Actors

- A1. 新项目/空项目用户：打开欢迎页后，需要快速选择从素材或故事开始。
- A2. 回访用户：每天进入时，希望感受到当日主题和轻量仪式感，但不希望入口变复杂。
- A3. 老黄历 API：为欢迎页提供宜忌、吉时、方位等真实农民历信息。
- A4. 后续实现者：需要明确哪些行为是第一版必须交付，哪些问题留到规划与实现阶段验证。

---

## Key Flows

- F1. 首屏进入
  - **Trigger:** 用户打开空项目或新项目欢迎页。
  - **Actors:** A1, A2
  - **Steps:** 页面根据当天北京时间确定纳音主题；渲染对应插画、动画和基础日期信息；两个开始按钮保持可见；用户选择其中一个入口。
  - **Outcome:** 用户能在不阅读黄历细节的情况下开始工作。
  - **Covered by:** R1, R2, R4, R5, R12

- F2. 展开今日气息
  - **Trigger:** 用户主动展开欢迎页的农历/老黄历细节层。
  - **Actors:** A1, A2, A3
  - **Steps:** 页面展示农历、干支、纳音和老黄历 API 返回的宜忌、吉时、方位等信息；用户可以收起该层并回到两个开始入口。
  - **Outcome:** 页面提供每日仪式感，但不抢占主要开始路径。
  - **Covered by:** R2, R7, R8, R9

- F3. 老黄历 API 不可用
  - **Trigger:** 页面加载时 API 请求失败、超时、额度耗尽或返回字段不完整。
  - **Actors:** A1, A2, A3
  - **Steps:** 页面继续显示本地可计算的农历、干支和纳音；隐藏或降级展示缺失的老黄历扩展内容；两个开始按钮继续可用。
  - **Outcome:** API 问题不会阻断欢迎页，也不会阻断用户开始工作。
  - **Covered by:** R7, R10, R12

---

## Requirements

**核心入口与结构**
- R1. 欢迎页必须采用两层结构：首屏是简洁欢迎入口，第二层是可展开/可收起的“今日气息”信息。
- R2. 首屏必须保留两个主要按钮，文案分别为“上传素材开始”和“聊一个故事开始”。
- R3. 两个按钮必须始终是首屏的主要行动入口；农历、纳音和老黄历信息不能喧宾夺主。
- R4. 点击“上传素材开始”应进入素材上传路径；点击“聊一个故事开始”应进入故事聊天路径。
- R5. 页面在桌面和移动视口下都不能出现按钮、文案、插画或展开层互相遮挡的问题。

**纳音五行与视觉表现**
- R6. 页面主题必须根据当天北京时间对应的纳音五行自动变化。
- R7. 页面应展示当天的基础日历身份，包括公历日期、农历日期、干支、纳音名称和五行。
- R8. 页面应保留用户提供的 standalone HTML 参考稿中的五行饮品插画与动画方向，包括金/木/水/火/土对应的饮品意象和轻量动效。
- R9. 美术上允许为了适配当前产品做简化、裁切、响应式调整和细节清理，但不能大幅改变参考稿的视觉方向或替换为全新的风格。

**老黄历信息**
- R10. 第一版必须接入真实老黄历 API，用于提供宜忌、吉时、方位等农民历扩展信息。
- R11. 老黄历 API 返回的信息应放在第二层“今日气息”中展示，而不是挤占两个开始按钮的主视觉位置。
- R12. 当 API 字段缺失时，页面应只展示可用字段，不能为了凑满版面而伪造权威黄历信息。
- R13. 当 API 请求失败或不可用时，页面必须优雅降级，保留本地可计算的农历、干支、纳音和五行主题，并保证两个开始按钮正常使用。

**内容语气**
- R14. 文案应保持 Drinking Time 的创作陪伴感和饮品隐喻，不应变成严肃算命、强运势判断或权威命理口吻。
- R15. 老黄历内容可以增加仪式感和创作灵感，但不能暗示系统会根据黄历替用户决定创作路径。

## Acceptance Examples

- AE1. **Covers R1, R2, R3, R4.** Given 用户打开空项目欢迎页, when 页面加载完成, then 用户首先看到“上传素材开始”和“聊一个故事开始”两个主按钮，并能点击进入对应路径。
- AE2. **Covers R6, R7, R8, R9.** Given 今天的纳音五行是某一元素, when 用户打开欢迎页, then 页面显示该元素对应的饮品插画、动画氛围、纳音名称和五行信息，且整体视觉仍能看出来自用户提供的参考稿。
- AE3. **Covers R10, R11, R12.** Given 老黄历 API 成功返回宜忌和吉时信息, when 用户展开“今日气息”, then 用户能看到这些 API 信息，但首屏两个开始按钮不被挤走或弱化。
- AE4. **Covers R13.** Given 老黄历 API 请求失败, when 用户打开欢迎页, then 页面仍显示本地农历/干支/纳音主题，老黄历扩展信息降级展示或隐藏，两个开始按钮仍可点击。
- AE5. **Covers R14, R15.** Given 老黄历信息显示在页面上, when 用户阅读文案, then 文案应像创作提示或每日氛围，不应像强制命理建议。

---

## Success Criteria

- 用户打开欢迎页后，能在 3 秒内理解两个开始路径，并能立即开始上传素材或聊天。
- 欢迎页每天有可感知变化，且变化来自纳音五行、农历和老黄历信息，而不是随机装饰。
- 用户提供的参考 HTML 中的插画和动画风格被明显保留下来，没有被重做成另一套视觉语言。
- 老黄历 API 成功时能增加页面信息价值；失败时不影响核心产品路径。
- 后续 `ce-plan` 可以直接基于本文档拆解实现，不需要重新决定首屏结构、视觉保留程度、API 是否接入或降级行为。

---

## Scope Boundaries

- 不重做整个分析工作台、镜头表、故事 Agent 或素材上传流程。
- 不手写完整权威老黄历算法；第一版依赖老黄历 API 获取扩展黄历信息。
- 不新增星座、生肖运势、社交分享、个人命盘、复杂偏好设置或用户画像。
- 不做大规模美术重绘，不替换用户提供参考稿里的五行饮品视觉方向。
- 不让老黄历内容替代或弱化“上传素材开始”“聊一个故事开始”两个核心入口。

---

## Key Decisions

- 两层结构：首屏保持开始路径清晰，第二层承载农历和老黄历信息，避免仪式感压过任务入口。
- 第一版接入真实老黄历 API：用户明确希望第一版就使用 API，而不是先用氛围规则模拟宜忌内容。
- 保留参考稿插画与动画：视觉改造应以迁移、适配和整理为主，不重新设计一套欢迎页美术。
- 老黄历失败时降级而不是阻断：欢迎页的主要价值仍是开始创作，外部 API 不能成为入口可用性的单点故障。

---

## Dependencies / Assumptions

- 需要在规划或实现阶段选择具体老黄历 API 服务商，并确认其字段、限额、授权、费用、归因要求和稳定性。
- 需要确认 API 能否提供本需求关心的宜忌、吉时、方位等字段；若某些字段不可得，应在实现计划中调整第一版展示范围。
- 日期计算应以北京时间为准，避免用户设备时区导致当天主题与农历信息不一致。
- 用户提供的 standalone HTML 参考稿是视觉与动效方向的来源；实现时可做产品适配，但不应把它当作一次全新设计机会。
