# 情绪处理逻辑：情绪的流动 → Shot Production Table 提示词

## 摘要

把情绪逻辑理成**三层**——**情绪源**（情绪卡，如实镜像）→ **情绪流动**（剧本情感弧线 + 镜头 beat）→ **关系型提取**成每镜出图提示词（带「在弧线里的位置 + 与上一镜的情绪转变」）；并把 Shot Production Table 定为后端 `shots` 表的**真·前端映射**，每镜的情绪与提示词都落在 DB 里。核心守则：**流动从用户真实弧线读出，不为戏剧性放大、不制造转折、不翻负。**

## 问题（Problem Frame）

现在情绪在管线里是「散落」的：情绪卡有情绪、剧本有情感弧线、镜头有 `mood`/`emotion`/`beat`，但**情绪没有被系统地、带着「流动」地写进每镜的出图提示词**——`promptDraft` 主要由视觉字段（主体/动作/场景）拼成，情绪大多只塞进 `mood` 字段，出来的图既不带情绪、更不带「情绪的转变」。同时前端 Shot Table（Creation 侧）是从 `storyShots` **前端合成的假数据**（假 id），不是后端 `shots` 表的真映射，前后端对不齐。结果：产品最核心的「转瞬即逝的情绪」，在落到画面这一步被丢掉了。

## 角色（Actors）

- **A1 创作者**：讲一段有起伏的小事，期待成片画面真的带着他情绪的「流动」（起 → 转 → 落），而不是一张张静态情绪图。
- **A2 情绪提取逻辑（系统）**：把情绪卡 + 情感弧线 + beat，**关系型**地提取成每镜的情绪电荷。
- **A3 审美层**：把情绪电荷翻成具体视觉语言（光/色/构图）写进 `promptDraft`。
- **A4 Shot Production Table**：后端 `shots` 表的前端映射，承载每镜的情绪与提示词（单一真相源）。

## 关键流程（Key Flows）

- **F1 情绪源 → 流动**：情绪卡的情绪 + 剧本情感弧线 + 镜头 beat，合成出「每镜在弧线里的位置 + 与相邻镜的转变」。
- **F2 流动 → 提示词**：每镜的情绪电荷 =「自身情绪 + 流动（位置/转变）」；审美层把它翻成视觉语言写进 `promptDraft`。
- **F3 落库**：每镜（含情绪字段 + `promptDraft`）写进后端 `shots` 表；前端 Shot Table 读它（真映射）。

## 需求（Requirements）

- **R1 情绪源如实镜像**：情绪卡的情绪（主/混合）忠于用户表达，不强行翻负。
- **R2 情绪流动建模**：用剧本情感弧线 + 镜头 `beat`（开场/起势/转折/收束）表示情绪如何移动（起/转/落）。
- **R3 关系型提取**：每镜出图提示词所带的情绪电荷，由「这一镜自身情绪 + 它在弧线里的位置 + 与上一镜的情绪转变」共同决定；**转折镜要表达「转变」本身**，而非一张静态情绪图。
- **R4 审美层翻译**：情绪电荷由审美层翻成具体视觉语言（光影/色彩/构图等）写进每镜的 `promptDraft`。
- **R5 守则——流动 ≠ 戏剧化**：情绪流动从用户**真实**弧线读出，不为戏剧性放大、不制造转折、不翻负（沿用「如实镜像」原则）。
- **R6 Shot Table = 后端 DB 真映射**：前端 Shot Production Table 是后端 `shots` 表的映射；每镜的情绪字段与 `promptDraft` 落在 DB（单一真相源），前后端一致。
- **R7 现状缺口修复**：Creation 侧当前从 `storyShots` 前端合成（假 id）的 Shot Table，改为读真后端 `shots`。

## 验收示例（Acceptance Examples）

- **AE1（覆盖 R3）**：一段「温暖 → 失落」的小故事，转折镜的出图提示词带「从暖到冷的转变」，画出的图体现**转变**，而不是一张静态悲伤图。
- **AE2（覆盖 R5）**：用户讲的是平淡温和的故事，系统不无中生有地制造转折、不翻负；每镜情绪如实。
- **AE3（覆盖 R6、R7）**：在前端改一镜，落到后端 `shots` 表；刷新/换页后一致；Analysis 与 Creation 看到同一份 shots。
- **AE4（覆盖 R4）**：同一镜内容，情绪电荷不同（暖 vs 冷），`promptDraft` 里的光/色/构图描述随之不同。

## 成功标准（Success Criteria）

- 成片画面带着情绪的「流动」（起伏/转变），不是静态情绪图的拼接。
- 用户真实平淡的故事不会被强行戏剧化。
- Shot Table 前后端对齐，情绪与提示词有单一真相源。

## 范围边界（Scope Boundaries）

- 跨项目情绪/审美记忆（v1 项目内即可）。
- 更换情绪分类模型/算法（本次是逻辑与数据结构的梳理，不换模型）。
- 手机端（单独重设中）。
- 美术 Agent 视觉锚（已单独成需求文档；此处只保证下游「吃情绪」的逻辑与之对齐）。

## 关键决策（Key Decisions）

- **关系型/流动驱动提示词（非静态 per-shot）。** 理由：对得上产品核心「情绪的流动」；让转折镜画出「转变」。
- **流动 ≠ 戏剧化（守则）。** 理由：守住「如实镜像」护城河，不背叛「把表达权还给用户、不替他放大」。
- **Shot Table 是后端 `shots` 表的真映射（单一真相源）。** 理由：前后端对齐，情绪/提示词可持久、可复用。
- **本次产出是 spec（逻辑 + 数据要求）**；实现 / 数据层改造留给 planning。

## 依赖与假设（Dependencies / Assumptions）

- 依赖现有：情绪卡（`StoryCard` 的 `emotion`/`emotionBlend`）、剧本情感弧线（`arc`/`arcSummary`）、镜头 `beat`、后端 `shots` 表（已有 `mood`/`promptDraft` 列）。
- 假设：现有 `shots` 表字段足以承载情绪电荷 + 关系型 prompt（可能需小幅加字段，planning 定）。
- 依赖：与「美术 Agent 视觉锚」需求在「下游如何把 **情绪 + 视觉** 综合成一镜 prompt」上对齐。

## 待解问题（Outstanding Questions，留给 planning）

- 关系型情绪电荷的具体表示：存一个「转变 delta」字段？还是出图时按相邻镜实时算？
- 提取在哪发生：剧本合成时（`synthesizeShotList`）写入，还是出图前算？
- Shot Table 真映射的迁移：Creation 侧从 `storyShots` 合成 → 改读真 `shots` 表，数据迁移/双写策略。
- 情绪电荷 与 美术 Agent 的「视觉锚」如何在同一镜的 prompt 里合成（两条需求的交汇点）。
