# 故事是唯一单位：镜头按 storyId 归属

## 问题

一个 project 可挂多个 story，但各界面解析"当前故事"的方式不一致：
- Story 页认用户显式打开的 `activeStoryId`（卡片/剧本/镜头读 `story.body`）。
- creation 聊天用 `getLatestStoryForProject`（项目里最近更新的故事）。
- Shot Production Table（`shot.list`）只认 `projectId`，不绑 storyId。

于是同 project 多故事时三者错位，用户看到"聊天、卡片、剧本、镜头表说的不是一个故事"。

## 现在的模型（务必遵守）

**故事（story）是唯一工作单位，项目（project）只是文件夹。**

- 镜头表 `shots` 有 `storyId` 列，镜头按 storyId 归属。
- 唯一真相源是 `shots` 表（按 storyId），**不是** `story.body.shots`。body.shots 仍存在但不再作 Creation 侧镜头来源（回填时只作"该故事应有几条"的参照）。
- "当前故事" `activeStoryId` 住在 `client/src/features/analysis/hooks/useProjectData.ts`（per-project 分槽持久化 `dt:activeStoryId:{projectId}`）；StoryAgentContext 通过 `onActiveStoryChange` 向上 emit，Creation 侧（shot.list + creation 聊天）都跟随它。
- `getLatestStoryForProject` 已删除——不要再用"取最新故事"做故事来源。

## 安全铁律

**所有按 storyId 的镜头查询/写入必须带 `userId` 条件。** 旧的 `getProjectShots` 不按 userId 过滤；改 storyId 后若不加 userId，等于"猜 storyId 取他人镜头"。
- `getStoryShots(storyId, userId)` / `replaceDirectorShotsForStory(storyId, userId, rows)` 两路径（memory + MySQL）都强制 userId。
- 端点（creation 聊天 / inpaint / 剧本合成 classify）接 storyId 后第一步 `getStoryById(storyId, ctx.user.id)` 验归属，再用。参照 `routers.ts` 的 generateForMobile 模式。
- `replaceDirectorShotsForStory` 必须保留 `intentType === "director_note"` 过滤，否则会误删该故事的非导演镜头。

## 数据迁移的坑（回填镜头 storyId）

- **不能用 shotNo 精确匹配**：`shots.shotNo` 是 "SH01" 字符串、`story.body.shots[].shotNo` 是数字 1，编号体系不兼容，精确匹配恒为空。
- **"归最近更新的故事"是错误兜底**：会系统性归给空壳新故事，丢失真实归属。
- 正确做法（`scripts/backfill-shot-storyid.ts`）：按 `body.shots.length` 与该 project 待归属镜头数**最接近**归属，数量并列/无法区分时标"歧义、拒绝自动落盘"，交用户 `--override projectId:storyId` 裁决；写前备份；userId 不一致不跨用户污染。

## 已知缺口

- `generated_images` 仍按 `projectId + shotNo` 关联图片：同 project 多故事同 shotNo 会串图。本次未收敛图片关联键，需要时单独把图片也归到 storyId。
